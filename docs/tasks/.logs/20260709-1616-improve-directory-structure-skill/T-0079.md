# T-0079 — Generic-code edge extractor wired to the scanner core

## Task

Wire its output into the T-0077 analysis core so the same scanner CLI can target a plain code repo, not just claude-code-shared.

## Implementation

Created `claude-code-shared/scripts/generic_code_extractor.py`:
- `discover_source_files()` walks an arbitrary directory for .js/.jsx/.ts/.tsx/.py files, skipping node_modules/.git/__pycache__/dist/build/venv/.venv/dot-dirs.
- One `file:<relpath>` node per source file, cluster = containing dir (or "root").
- JS/TS edges: `import ... from '<path>'`, bare `import '<path>'`, `require('<path>')`, `export ... from '<path>'` -- only relative (./ or ../) specifiers resolved against on-disk files, trying literal path, .js/.jsx/.ts/.tsx suffixes, and index.* files.
- Python edges: `import <module>` and `from <module> import ...`, including relative imports (`from . import x`, `from .foo import x`) -- dotted paths mapped to on-disk .py/__init__.py candidates relative to root (absolute) or the importing file's package (relative). Unresolvable (stdlib/third-party) imports silently skipped.
- Per-file exceptions caught so a single malformed file never aborts the walk.
- metrics: fan_in/fan_out derived from edges, dir_depth from path, orphan flag, optional churn via `git log --name-only` (best-effort, silently empty if not a git repo).
- `build_graph(base_dir)` usable as a library import; standalone CLI via `python3 generic_code_extractor.py [base_dir] [--out ...]`.

Wired into `claude-code-shared/scripts/scanner.py`:
- Imported `generic_code_extractor` alongside `claude_tooling_extractor`.
- Registered `"generic-code": generic_code_extractor.build_graph` in `EXTRACTORS`.
- `default_root_for` already fell through to `os.getcwd()` for any non-"claude-tooling" extractor, so generic-code gets cwd-default for free -- no change needed there beyond the docstring update.
- Updated the module docstring's "## Extractors" section to document generic-code.

## Manual verification (RED/GREEN loop, per infra convention -- no pytest)

Built a throwaway fixture at `scratchpad/generic-code-fixture/` with:
- `js/index.ts` importing `./utils/math` (named export), `./utils/helper` (default/require-style), `./utils/legacy` (via `require(...)`), plus a bare `some-package` import (expected to be skipped).
- `py/pkg/main.py` importing `os` (stdlib, expected skip), `from pkg.helper import do_it` (absolute dotted), `from .sibling import value` (relative), `from .sub.thing import value` (relative dotted) -- plus `py/pkg/__init__.py`, `py/pkg/sub/__init__.py` as unimported/orphan packages.

Commands run and results:

1. `python3 scanner.py --extractor generic-code <fixture>` (whole fixture, js+py together): 10 nodes, 5 edges. JS edges all 3 resolved correctly (math.ts, helper.js, legacy.js); bare `some-package` correctly skipped. Python `from pkg.helper import do_it` did NOT resolve here because root is the fixture root (containing both js/ and py/), so the absolute dotted path `pkg.helper` doesn't match on-disk relative to that root -- expected/correct behavior for an absolute import outside its true package root, not a bug.

2. `python3 scanner.py --extractor generic-code <fixture>/py` (py/ as root, so `pkg` is the true top-level package): 6 nodes, 3 edges -- `main.py -> helper.py`, `main.py -> sibling.py`, `main.py -> sub/thing.py`. All three resolve exactly as expected, confirming absolute vs relative import resolution both work correctly when root aligns with the package root.

3. Piped both graphs into `dependency-graph-findings.py`: both parsed with 0 validation errors; `summary` counts (nodes/edges) matched the extractor output exactly; 0 findings (expected -- clean small fixture, no litmus violations).

4. Strengthened the check: edited `sibling.py` to add `from .main import run`, creating a genuine main.py <-> sibling.py import cycle. Re-ran extractor + core: edges now include `sibling.py -> main.py`; findings now include exactly one ADP finding: "Cycle in edge graph: file:pkg/main.py -> file:pkg/sibling.py -> file:pkg/main.py" -- confirms the wiring surfaces findings consistent with the fixture's actual (deliberately introduced) import structure, not just an empty pass-through.

5. Malformed-file robustness: wrote a `.py` file with invalid UTF-8 bytes (`broken.py`). Extractor completed without crashing; the file was still discovered as a node (file:pkg/broken.py) but contributed no edges (import extraction silently skipped on decode error), confirming per-file exception handling. Removed the file after the check.

6. Standalone module CLI (`python3 generic_code_extractor.py <dir>`) and no-root-arg cwd-default (`cd <dir> && python3 scanner.py --extractor generic-code`) both verified working, matching the claude_tooling_extractor.py CLI conventions.

7. Regression check: `python3 scanner.py --extractor claude-tooling` (no root arg) still runs cleanly against claude-code-shared itself: 100 nodes / 204 edges (100 vs T-0078's reported 99 is expected -- generic_code_extractor.py is a new file under scripts/, counted as one additional script node; not a regression).

8. `python3 scanner.py --list-extractors` correctly lists both `claude-tooling` and `generic-code`.

All acceptance criteria met:
- [x] `scanner --extractor generic-code` against a sample JS/TS or Python directory produces nodes/edges/metrics JSON with no parse errors.
- [x] Piping that output into the T-0077 core produces findings consistent with the sample directory's known import structure (0 findings on the clean fixture; exactly 1 correctly-identified ADP cycle finding after a deliberate cycle was introduced).
