# T-0080 — Scanner --check mode for reference-integrity validation

Start: 2026-07-09

## Task
Add a --check mode to the scanner CLI reusing the edge-resolution pass from T-0078/T-0079 to assert zero dangling edges and no new cycles, exiting non-zero on failure. Must follow .cch/.cco symlinks (not treat as ordinary edges), and detect literal absolute-path string references in addition to structured registry entries.

## Acceptance criteria
- [ ] `scanner --check` against current claude-code-shared repo exits 0 with no dangling edges reported (NOTE: expected to legitimately find the pre-existing 'run-tasks' dangling ref per T-0078 breadcrumb — verify --check surfaces it, non-zero exit is correct/expected here)
- [ ] Deliberately introducing a dangling reference and re-running --check causes non-zero exit naming the broken edge (manual verification, no unit test per infra convention)

## Implementation notes

Per project convention (no unit tests for infra work), this task used
manual-verification cycles instead of /tdd's automated red-green-refactor loop.

### Design decisions

1. **--check as a flag on scanner.py**, not a separate script. Defaults
   --extractor to claude-tooling when --extractor is omitted (matches literal
   `scanner --check` invocation in acceptance criteria).

2. **claude_tooling_extractor.check_integrity(base_dir)** reuses the exact
   same discovery/resolution helpers as build_graph (discover_skills,
   load_registry, discover_scripts, resolve_script_path, owning_node_id) but
   inverts build_graph's documented silent-skip behavior: anything that fails
   to resolve is recorded as a dangling-reference issue instead of dropped.
   Covers: registry.json consumers + file-existence, script-path refs in
   skill/agent markdown, model-tiers.json keys, skill-pipeline.json stage
   links, plus a new literal absolute-path scan (signal 6).

3. **Literal absolute-path scan (ALT_ROOT_RE)**: matches `claude-code-shared/<path>`
   (covers both `~/.dotfiles/claude-code-shared/...` and the fully-resolved
   `/Users/<u>/.dotfiles/...` form -- substring survives either way) and
   `.cch|.cco/(skills|agents|settings.json)<path>` specifically -- NOT a
   blanket `.cch/.cco` alias, since ~/.cch and ~/.cco are Claude Code's
   per-profile home dirs with lots of unrelated content (session transcripts
   under `.cch/projects/...`, etc.). Per CONTEXT.md invariant 4, only
   skills/, agents/, and settings.json are symlink-farmed from `.cch`/`.cco`
   into claude-code-shared -- confirmed live via `readlink -f ~/.cch/skills`
   -> `/Users/eric/.dotfiles/claude-code-shared/skills`.

4. **Scan scope for signal 6** restricted to authored/structural content
   (CHECK_SCAN_DIRS = skills, agents, scripts, resources, contracts, hooks,
   plus CONTEXT.md/settings.json/skill-pipeline.json at root). Deliberately
   excludes learnings/ and usage-reports/ (runtime-generated data: session
   transcript paths, audit logs, historical schema tags) which produced
   9 false positives on first pass (learnings/unified-learnings.jsonl
   references to old .cch/.cco project-transcript paths, hooks/stop-hook.py's
   documented-but-not-yet-created audit log path). Also excludes any target
   ending `.jsonl` as a second guard (log/data format, not authored ref).

5. **registry_consumer check widened**: a `consumers` entry is only dangling
   if it matches neither a skill on disk NOR a registered agent name.
   Discovered mid-verification that export-tasks-gh/export-tasks-notion list
   'export-tasks' (the coordinator *agent*, not a skill) as their consumer --
   a legitimate agent-to-agent spawn relationship build_graph's edge model
   doesn't currently represent, but not a broken reference.

6. **Cycle detection excludes pipeline_next edges**. dependency-graph-findings.py's
   ADP (Acyclic Dependencies Principle) check ran against the full graph and
   found 2 cycles: dispatch-tasks -> to-e2e-tasks -> dispatch-tasks, and
   prototype -> to-seed -> prototype. Both are skill-pipeline.json's
   next-skill *workflow suggestions*, which loop back by design (e.g.
   generated e2e tasks get dispatched again; a seed captured from prototyping
   can send you back to prototype more) -- not structural dependency cycles.
   Verified by filtering pipeline_next out of the edge list before piping
   into dependency-graph-findings.py: 0 cycles remain. --check now excludes
   pipeline_next edges from cycle detection so it isn't permanently red for
   an intentional design property of the skill pipeline.

7. **Cycle detection reuse**: invoked dependency-graph-findings.py as a
   subprocess (piping the graph JSON via stdin, same pattern documented in
   its own module docstring) rather than re-implementing DFS cycle-detection
   in scanner.py or importing it (its filename has hyphens, not a valid
   Python module name).

## Closing summary

Added `--check` (reference-integrity mode) to scanner.py, defaulting to the
claude-tooling extractor. It reuses claude_tooling_extractor's existing
discovery/resolution helpers via a new `check_integrity()` function that
inverts build_graph's silent-skip-on-unresolved behavior into recorded
dangling-reference issues (registry.json consumers + file existence, script
refs, model-tiers.json keys, skill-pipeline.json links, and a new literal
absolute-path scan that correctly follows the `.cch`/`.cco` symlink-farm
aliasing documented in CONTEXT.md invariant 4 rather than treating those
paths as broken). Cycle detection reuses dependency-graph-findings.py's ADP
check via subprocess, with pipeline_next edges excluded since those
represent intentional workflow loops, not structural defects. Verified
against the live repo (correctly surfaces the pre-existing FU-001 'run-tasks'
stale consumer, exit 1) and against a scratch fixture with a deliberately
broken registry entry (correctly named, exit 1; real repo untouched).
