Problem Statement

The grill skills (grill-me, grill-with-docs) offer "Create a task file" as a direct session-end option, bypassing the PRD step that structures decisions for downstream consumption. This creates three problems.

First, tasks generated from raw conversation context lack the structured user stories and implementation decisions that make them executable. to-tasks was designed to consume a PRD, not a conversation.

Second, to-prd-html requires all sections to have content. Architecture decisions, infrastructure changes, and deployment planning don't produce user stories in the feature-delivery sense. Running to-prd-html after these sessions renders PRDs with empty blocks that mislead reviewers and downstream tools.

Third, the browser_verify and dynamic route guidance in to-tasks SKILL.md is ~245 words, carrying redundant prose that bloats the skill without adding rule clarity.

Solution

Enforce a single workflow chain: every grill session that produces actionable output routes through a PRD before reaching tasks. Update the grill skills' session-end protocols to offer "Create a PRD" instead of "Create a task file." Fix to-prd-html to render only populated sections. Tighten the browser_verify block in to-tasks SKILL.md in place.

Evidence: The /improve-skill to-tasks eval scored 92.8/100 behavioral. The no-PRD scenario and a live grilling session both surfaced the direct grill-to-tasks gap as a real workflow problem.

Success metrics:

  • Grill session-end no longer presents "Create a task file" in either grill skill
  • to-prd-html renders zero empty section blocks when given non-feature input
  • to-tasks /improve-skill behavioral score holds at 92 or above after SKILL.md tightening
  • browser_verify block in to-tasks SKILL.md reduced from ~245 words to 170 or fewer

The four-skill chain after this change. Each arrow is a session boundary: a durable file artifact is written before the boundary, enabling safe session kills and restarts at any step.

Alternatives Considered

to-objectives skill (dropped)

A lightweight intermediary skill that extracts structured decisions from any conversation into the extract-prd-json shape. Would produce docs/objectives/*.json consumed by to-tasks.

  • Faster, lower token cost than a full PRD
  • No document artifact to review or share
  • Divergent codepath in to-tasks (two input types)
  • New skill to build and maintain
  • Single workflow chain broken into two variants

Always use PRD (chosen)

Single artifact type. Fix to-prd-html optional sections so it handles any conversation type. One workflow, one chain.

  • One artifact type, no divergent workflow
  • Reviewable and shareable document
  • to-tasks needs zero changes
  • Slightly more token cost
  • Requires fixing to-prd-html optional sections

Extract browser_verify block

Move ~245 word block to resources/browser-verify-rules.md. Model reads a reference file to get the rules.

  • Passes structural token metric
  • Model may skip the reference read
  • Risk of browser_verify rule misses in practice
  • Currently scoring 100% behavioral — extract risk not justified

Tighten in place (chosen)

Cut redundant prose within SKILL.md. Keep all rules inline. Target 170 words or fewer.

  • Rules stay inline, no reference hop
  • Modest token reduction
  • Structural metric still flags it (over 200 words even tightened)
  • Behavioral score preserved

to-objectives skill — dropped in favor of always-PRD approach

Further Notes

Session isolation model. Each step in the chain writes a durable file artifact before the session boundary. Safe kill points: after grill session ends and to-prd-html writes the HTML file; after to-tasks writes the JSON file; after any completed run-tasks task. If a session fills mid-step, use /handoff to continue in a new window.

Handoff vs. chain. /handoff is for mid-work continuity when the session fills before a step completes. The chain skills are for moving forward through completed steps. They don't overlap — handoff preserves the reasoning thread, the chain skills consume completed artifacts.

Structural score tradeoff. to-tasks structural score stays at 71/100 (5 of 7 items passing). The two failing items are intentionally left as-is because the inline content is load-bearing for behavioral quality. Structural metrics are advisory.

Risks.

  • Tightening browser_verify prose risks subtle rule loss. Run /improve-skill to-tasks immediately after to catch any regression before merging.
  • Removing "Create a task file" from grill skills changes muscle memory. Users who relied on grill to tasks directly must now route through a PRD step.
  • to-prd-html optional sections require careful template logic. Fully-populated PRDs must continue rendering correctly.
