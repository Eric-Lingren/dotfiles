Problem Statement

Every conversation, grill-me session, and grill-with-docs session needs a durable outcome. This happens dozens of times a day. Today the only forward path from a grill is to-prd-html, which synthesizes a full interactive HTML PRD. The most recent such run cost roughly six minutes and about 25,000 tokens for a single document.

The HTML is genuinely useful for human review and sharing. It is not, however, required by the skills that consume the work. to-tasks already discards the HTML and reads only the embedded prd-data JSON via extract-prd-json.sh. The beautified document is a human view layer, not a machine dependency.

Two structural problems follow. First, synthesis (turning a messy conversation into structured decisions) happens inside to-prd-html and is never reused; any later consumer re-derives it. Second, the workflow forces a heavy PRD on every run even when no shareable document is needed, and even for non-feature conversations (architecture, bug investigation, exploration) that produce no user stories.

The existing handoff skill is the only lightweight alternative, but it writes free-form markdown prose with no fixed schema. It is ephemeral and non-deterministic, unsuited to feeding downstream skills.

Solution

Introduce to-seed: a deterministic skill that distills any conversation or grill session into a single standardized JSON artifact, the intermediate representation (IR). Synthesis happens exactly once, in to-seed. Every other skill consumes the JSON.

to-prd-html is refactored into a pure renderer (JSON in, HTML out). to-tasks consumes the JSON through the same normalizer it already uses. The seed uses the existing prd-data shape extended with a deterministic core (summary, decisions, open_threads, next_action) and a schema_version, with all feature-specific limbs optional. One standardized format means one ingestion path and no divergent codepaths.

The seed is cheap, raw, and machine-first. The HTML PRD becomes an optional view, rendered from a seed only when a human needs to review or share.

Evidence: The motivating to-prd-html run produced one PRD in about six minutes at roughly 25,000 tokens. to-tasks already ignores that HTML and extracts only the embedded prd-data JSON, proving the document is a view layer rather than a machine dependency.

Success metrics:

  • A to-seed run costs materially less than a to-prd-html run on the same conversation, since the HTML emit, the context-loader hop, and three interactive prompts are removed.
  • to-tasks ingests a to-seed .json with zero changes to its ingestion call; extract-prd-json.sh normalizes .html, .md, and .json to the same prd-data JSON.
  • to-prd-html, as a pure renderer, produces zero empty section blocks when given a partial seed.
  • to-seed scores fidelity-dominant on /improve-skill: no fabricated fields, correct decided-vs-open separation, no dropped decisions, across feature, architecture, bug, and exploratory scenarios.
  • A malformed seed fails at extract-prd-json.sh validation, not inside to-tasks.
Architecture

The JSON seed is the intermediate representation. One synthesizer produces it. Every consumer reads it through one normalizer. The HTML PRD is one of several optional consumers, not a required stop.

Boundary contract: extract-prd-json.sh normalizes .html, .md, and .json inputs into the one prd-data JSON shape and validates it against prd-data-schema.json. Consumers ignore unknown fields, so the schema can grow additively under schema_version.

Alternatives Considered

Always PRD (prior PRD's choice, now reversed)

Route every grill through a full HTML PRD before tasks. One artifact type.

  • One artifact, reviewable and shareable
  • to-tasks needed zero changes
  • Forces ~25k-token HTML synthesis on every run
  • Breaks on non-feature conversations (no user stories)

Seed IR + optional PRD (chosen)

Synthesize once into cheap JSON. HTML becomes an optional render.

  • Cheap, deterministic, machine-first
  • One ingestion format keeps to-tasks single-path
  • Handles any conversation type via optional limbs
  • Requires the to-prd-html renderer refactor + normalizer

to-prd-html dual-mode

Keep synthesize-from-conversation as a fallback alongside the renderer.

  • No behavior removed
  • Duplicates synthesis logic
  • Risks drift between "PRD from conversation" and "PRD from seed"

to-prd-html pure renderer (chosen)

One synthesis source (to-seed); to-prd-html only renders JSON.

  • Single synthesis truth, no drift
  • Cheap and deterministic render
  • Loses standalone "PRD straight from a chat" (acceptable: seed first, then render)

Name by format: to-json

Name the skill after the file format it emits.

  • Instantly signals raw machine data
  • Names the format, not the purpose
  • Lies if the format ever changes; collides with other JSONs
  • No intent in the session-end menu

Name by role: to-seed (chosen)

Name the skill after its role: the compact upstream source.

  • Captures "one source, many downstream things"
  • Survives a format change
  • Fits the to-X family; clear menu intent
  • Format stated in the description instead

Extract browser_verify to a file

Move the ~245-word block to a resources reference (prior PRD alt).

  • Passes the structural token metric
  • Model may skip the reference read
  • Behavioral risk not justified at 100% behavioral

Tighten in place (chosen)

Cut redundant prose within SKILL.md, keep all rules inline.

  • Rules stay inline, no reference hop
  • Modest token reduction
  • Behavioral score preserved

Auto-committing any artifact. to-seed never commits.
Further Notes

Determinism model. Artifact content is zero-prompt reproducible: the same conversation yields the same JSON. Output location gets exactly one human checkpoint, the branch sanity-check that leads with the current branch. These are two different axes and must not be conflated.

Model strategy. Start at sonnet/xhigh to protect distillation fidelity on the highest-value primitive, then scale down toward medium once the eval proves headroom. Re-register the tier when the model drops.

Schema evolution. schema_version governs forward compatibility. Consumers ignore unknown fields, so additive changes do not break old seeds.

Supersession. This PRD supersedes docs/prd/20260602-1922-grill-prd-tasks-chain.html. The old "always PRD" decision is reversed. Still-valid items from that PRD are folded in here: the to-prd-html optional-section rendering, the prd-data-schema optional fields, the to-tasks browser_verify tightening, the schema-block non-change, and the multi-branch out-of-scope note. The old file can be ignored.

Sequencing risk. Land to-seed, the schema update, and the normalizer before the grill menus change. Otherwise there is a window where grill session-end offers "Create a seed" but no skill exists to produce it.
